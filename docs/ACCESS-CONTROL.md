# Контроль доступа

Как ограничить круг людей, которые могут пользоваться Recall, и как закрыть доступ конкретному человеку.

**Установка:** открой Supabase → SQL Editor → New query → вставь весь `docs/schema.sql` → Run. Файл идемпотентный, повторный запуск безопасен.

---

## Как это устроено

Механизмов три, они независимы и закрывают разные вещи.

### 1. Белый список — кто может зарегистрироваться

Таблица `allowed_emails`. Регистрация в приложении остаётся открытой, но проходит только тот, чей адрес ты заранее внёс в список.

Работает это так. Когда Supabase создаёт пользователя, на таблице `auth.users` срабатывает триггер `handle_new_user` — он и раньше создавал профиль и две колоды. Теперь первым делом он проверяет email по списку и, если адреса там нет, бросает исключение. Триггер выполняется **внутри той же транзакции**, что и создание пользователя, поэтому исключение откатывает её целиком: записи в `auth.users` не остаётся, «полурегистрации» без профиля возникнуть не может.

Почему это нельзя обойти с клиента: проверка выполняется в базе, а не в браузере. Подменить запрос, поправить JS в DevTools или постучаться напрямую в REST API бесполезно — любой путь создания пользователя проходит через этот триггер.

Сама таблица `allowed_emails` закрыта наглухо: включён RLS и **не создано ни одной политики**, что в Postgres означает отказ во всём, плюс явный `revoke all`. Приложение не может ни прочитать список, ни даже проверить, есть ли в нём конкретный адрес — то есть перебором узнать, кого ты пригласил, невозможно.

Адреса нормализуются триггером: `Ivan@Mail.ru ` и `ivan@mail.ru` — одна и та же запись.

### 2. Флаг `blocked` — мягкая блокировка

Колонка `profiles.blocked`. Когда стоит `true`:

- приложение вместо всех экранов показывает «Доступ приостановлен» с кнопкой «Выйти» (`src/components/BlockedScreen.tsx`, проверка в `src/components/ProtectedRoute.tsx`);
- запросы к платному AI-прокси `/api/gemini` отклоняются с кодом 403 (`api/gemini.ts`, функция `authorize`).

Снять блокировку с себя пользователь не может: в схеме выполнены `revoke update on profiles` и `grant update (display_name, level, native_lang)`, то есть менять он вправе ровно три колонки, и `blocked` в их число не входит.

**Где граница этого механизма — важно понимать.** Флаг не отзывает уже выданный JWT. Заблокированный человек, если очень захочет, сможет через прямой REST-запрос читать *свои собственные* данные, пока не истечёт токен (обычно час). Чужие данные ему по-прежнему недоступны — за это отвечает RLS, который никак не связан с этим флагом.

То есть `blocked` — это «вежливо закрыть приложение и перекрыть расходы на AI», а не «выкинуть из системы».

### 3. Ban user — жёсткая блокировка

Встроенная функция Supabase: Authentication → Users → выбрать пользователя → Ban user.

Она мгновенно инвалидирует все токены: `/auth/v1/user` начинает отвечать ошибкой, и человек вылетает из приложения при первом же запросе. Это настоящая блокировка на уровне выдачи доступа, и именно её стоит использовать, если нужно отрезать кого-то всерьёз.

**Рекомендация:** для «поставил на паузу, скоро верну» — флаг `blocked`. Для «этот человек больше не должен сюда заходить» — Ban, при желании вместе с флагом (тогда, если бан снимут, доступ не откроется автоматически).

## Повседневные команды

Все выполняются в Supabase → SQL Editor.

### Пригласить человека

```sql
insert into public.allowed_emails (email, note)
values ('anya@example.com', 'подруга, тест июль');
```

Несколько сразу:

```sql
insert into public.allowed_emails (email, note) values
  ('anya@example.com',  'тестер 1'),
  ('boris@example.com', 'тестер 2');
```

Если адрес уже есть, вставка упадёт с ошибкой уникальности — это нормально. Чтобы молча пропускать дубликаты:

```sql
insert into public.allowed_emails (email, note)
values ('anya@example.com', 'тестер 1')
on conflict (email) do nothing;
```

### Посмотреть, что происходит

```sql
select * from public.access_overview;
```

Показывает по каждому приглашённому: зарегистрировался ли (`registered_at`), когда заходил последний раз (`last_sign_in_at`), заблокирован ли флагом (`blocked`), забанен ли в Supabase (`banned_until`), имя и роль.

### Заблокировать / разблокировать

```sql
-- заблокировать
update public.profiles set blocked = true
where id = (select id from auth.users where lower(email) = 'anya@example.com');

-- вернуть доступ
update public.profiles set blocked = false
where id = (select id from auth.users where lower(email) = 'anya@example.com');
```

Эффект наступает при следующей загрузке приложения у пользователя (или сразу — при следующем запросе к AI).

### Убрать из белого списка

```sql
delete from public.allowed_emails where email = 'anya@example.com';
```

Важно: на уже зарегистрированного это **не влияет** — список проверяется только в момент регистрации. Чтобы отрезать существующего пользователя, используй `blocked` или Ban.

### Удалить аккаунт полностью

Authentication → Users → … → Delete user. Каскад удалит профиль, колоды, карточки и всю активность.

## Что увидит пользователь

| Ситуация | Что видит |
|---|---|
| Регистрируется, адреса нет в списке | «Этот адрес не в списке приглашённых. Recall пока работает в закрытом тесте — напиши владельцу, чтобы он добавил твою почту.» |
| Регистрируется, адрес в списке | Обычная регистрация, дальше онбординг |
| Заходит с `blocked = true` | Экран «Доступ приостановлен» с кнопкой «Выйти» |
| Заходит после Ban user | Выбрасывает на страницу входа, войти не может |

На экране регистрации есть подпись «Recall сейчас в закрытом тесте — зарегистрироваться можно только по приглашению», чтобы отказ не был неожиданностью.

## Технические детали

**Почему текст ошибки приходится угадывать.** Supabase Auth не пробрасывает наружу текст исключения из триггера — при ошибке в БД GoTrue отдаёт обобщённое «Database error saving new user». Поэтому `describeSignUpError()` в `src/lib/access.ts` ловит и наш маркер `RECALL_NOT_INVITED`, и эту обобщённую формулировку. Всё остальное (слабый пароль, занятый email) пробрасывается как есть — иначе настоящие проблемы маскировались бы под «нет приглашения».

**Fail-open в проверках.** И `isBlocked()` на клиенте, и чтение профиля в `api/gemini.ts` при сетевой ошибке пропускают пользователя дальше. Это осознанно: запереть человека из-за отвалившегося интернета хуже, чем на минуту пустить заблокированного — тем более что жёсткая блокировка через Ban не зависит от этих проверок и срабатывает раньше, на валидации токена.

**Файлы, которых это касается:**

| Файл | Что делает |
|---|---|
| `docs/schema.sql`, блок «КОНТРОЛЬ ДОСТУПА» | Таблица, триггер-гейт, колонка `blocked`, view `access_overview` |
| `src/lib/access.ts` | Чтение флага, человеческие тексты ошибок |
| `src/components/BlockedScreen.tsx` | Экран «Доступ приостановлен» |
| `src/components/ProtectedRoute.tsx` | Проверка флага перед входом в приложение |
| `src/features/auth/LoginPage.tsx` | Показ понятной ошибки регистрации |
| `api/gemini.ts`, функция `authorize` | Отказ заблокированному в доступе к Gemini |

## Лимиты на AI

Отдельный механизм, но проверяется тем же вызовом. Без него любой приглашённый мог бы в цикле дёргать `/api/gemini` и за минуты сжечь дневную квоту Gemini — AI лёг бы сразу у всех.

**Текущие лимиты:** 40 запросов в час и 200 в сутки на пользователя. Живое занятие (диалог плюс разбор пары текстов) укладывается в 10–20 обращений, так что упереться при нормальном использовании трудно.

Счётчик — таблица `ai_calls`, закрытая от клиента полностью; пишется только через `consume_ai_quota()`. Подделать его из приложения нельзя.

Поменять лимиты — константы `max_per_hour` и `max_per_day` в начале функции `consume_ai_quota()` в `docs/schema.sql`, затем перезапустить файл.

Посмотреть расход:

```sql
select * from public.ai_usage_overview;
```

Сбросить лимит конкретному человеку прямо сейчас:

```sql
delete from public.ai_calls
where user_id = (select id from auth.users where lower(email) = 'anya@example.com');
```

**Что стоит знать:** квота списывается *до* обращения к Gemini, поэтому неудачный запрос (например, Gemini ответил 503) её всё равно тратит. Сделано намеренно — иначе счётчик можно было бы обходить, провоцируя ошибки.

## Чего здесь нет

**Самостоятельного управления из приложения.** Список ведётся из SQL Editor. Экран `/admin` сознательно не делался: ради списка из двух человек он потребовал бы новой роли, новых RLS-политик и нового экрана — то есть заметно расширил бы поверхность атаки без реальной пользы.
